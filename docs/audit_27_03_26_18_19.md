## 2. BUGS IDENTIFIÉS
 
### BUG CRITIQUE #1 — Chemins de revalidation obsolètes
**Fichiers:** `src/actions/accounting.ts`, `src/actions/donations.ts`, `src/actions/appointments.ts`
 
Les `revalidatePath()` pointent vers des routes `/portail/*` qui **n'existent plus** (l'app a migré vers `/admin/dashboard/*`). Conséquence : **le cache n'est jamais invalidé** après création de donations, écritures comptables ou rendez-vous.
 
```
revalidatePath("/portail/partenaire");   // MORT
revalidatePath("/portail/benevole");     // MORT
revalidatePath("/portail/tresoriere");   // MORT
```
 
**Correction :** Remplacer par `/admin/dashboard/dons`, `/admin/dashboard/compta`, etc.
 
### BUG CRITIQUE #2 — Password reset incomplet
**Fichier:** `src/app/api/auth/forgot-password/route.ts`
 
Le token de reset est généré (`crypto.randomBytes(32)`) mais **jamais stocké en base**. Il n'y a ni champ `resetToken`/`resetTokenExpiry` dans le modèle User, ni page `/reset-password?token=xxx`. La fonctionnalité est **non fonctionnelle**.
 
### BUG CRITIQUE #3 — Faille XSS dans les emails
**Fichier:** `src/lib/mailer.ts:93-99`
 
Les données utilisateur (`opts.name`, `opts.email`, `opts.content`) sont injectées directement dans le HTML de l'email **sans échappement**. Un attaquant peut injecter du HTML/JS via le formulaire de contact.
 
### BUG CRITIQUE #4 — XSS dans l'affichage d'articles
**Fichier:** `src/app/actualites/[id]/page.tsx:64`
 
```jsx
dangerouslySetInnerHTML={{ __html: article.content }}
```
 
Le contenu d'article est rendu en HTML brut. Si un admin est compromis ou si le contenu n'est pas sanitisé à l'entrée, c'est une faille XSS stockée.
 
### BUG #5 — Incohérence des rôles dans accounting.ts
**Fichier:** `src/actions/accounting.ts:13`
 
```typescript
if (!["SUPER_ADMIN", "DIRECTRICE", "BENEVOLE"].includes(role))
```
 
Un **BÉNÉVOLE** peut créer des écritures comptables ? C'est incohérent avec le middleware qui bloque `/compta` pour les bénévoles. De plus, seule "DIRECTRICE" est listée, pas les autres rôles Direction (DIRECTEUR, DIRECTION, TRÉSORIÈRE...).
 
### BUG #6 — Dashboard isDirection trop restrictif
**Fichier:** `src/app/admin/dashboard/page.tsx:14`
 
```typescript
const isDirection = role === "SUPER_ADMIN" || role === "DIRECTION";
```
 
Cela exclut DIRECTRICE, DIRECTEUR ADJOINT, TRÉSORIÈRE etc. du dashboard Direction. Ces rôles voient un **dashboard vide**.
 
### BUG #7 — Compta page accès trop restrictif
**Fichier:** `src/app/admin/dashboard/compta/page.tsx:13`
 
```typescript
if (!["SUPER_ADMIN", "DIRECTION"].includes(session.user?.role))
```
 
TRÉSORIÈRE et DIRECTRICE sont exclus de la page compta alors qu'ils devraient y avoir accès.
 
### BUG #8 — 10 suites de tests en échec
`@testing-library/dom` est manquant dans les dépendances. 10 suites de tests UI échouent.
 
### BUG #9 — 20 vulnérabilités npm
8 moderate, 12 high (picomatch ReDoS, rollup path traversal, undici HTTP smuggling).
 
### BUG #10 — Secret auth hardcodé
**Fichier:** `src/auth.config.ts:33`
 
```typescript
secret: process.env.AUTH_SECRET || "changeme_in_production_secret_key_123"
```

## 8. INTÉGRATION HIBOUTIK — PLAN DE SYNCHRONISATION
 
### Objectif
Connecter la caisse en ligne Hiboutik (API REST : `https://test.hiboutik.com/api/`) au portail Rescape pour synchroniser :
- **Produits/articles** de la boutique solidaire
- **Ventes** (recettes) vers la comptabilité Rescape
- **Stock** (dons reçus = entrées de stock)
- **Clients/adhérents** entre les deux systèmes
 
### Architecture proposée
 
```
[Hiboutik Cloud] <--API REST--> [Rescape Backend (Server Actions)]
                                        |
                                   [Prisma DB]
                                        |
                                [Portail Admin Rescape]
```
 
### Endpoints Hiboutik à utiliser
 
| Fonctionnalité | Endpoint Hiboutik | Synchronisation |
|----------------|-------------------|-----------------|
| Produits | `GET/POST /products` | Hiboutik -> Rescape (lecture) |
| Ventes | `GET /sales` | Hiboutik -> AccountingEntry (RECETTE auto) |
| Stock | `GET/PUT /stock` | Rescape Donation -> Hiboutik stock |
| Clients | `GET/POST /customers` | Rescape Membership <-> Hiboutik |
 
### Implémentation recommandée
1. **Nouveau modèle Prisma** `HiboutikSync` pour tracker les synchros
2. **Fichier `src/lib/hiboutik.ts`** — Client API avec authentification Basic Auth
3. **Server actions `src/actions/hiboutik.ts`** — Sync manuelle + auto
4. **Page admin `/admin/dashboard/caisse`** — Interface caisse avec tableau de bord synchro
5. **Cron job ou webhook** pour la synchronisation automatique des ventes
 
### Données à synchroniser
- **Ventes Hiboutik → Compta Rescape** : Chaque vente clôturée crée une `AccountingEntry` type RECETTE, catégorie VENTE
- **Dons Rescape → Stock Hiboutik** : Chaque donation matérielle met à jour le stock produit
- **Adhérents Rescape → Clients Hiboutik** : Membres avec carte synchronisés comme clients
